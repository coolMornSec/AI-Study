# AI 代理敏捷开发执行指南

本文件定义 AI 代理在本项目中的统一开发流程与交付约束。核心目标是在保障代码质量与规范的前提下，最大化 AI 代理的执行效率，彻底杜绝形式主义与 Token 浪费。

---

## 一、 核心协作准则 (Core Principles)

AI 代理在开发过程中**必须**严格遵循以下红线，任何违背将导致任务被打回：

1. **精准履约与边界意识**：只执行明确要求的任务。当发现严重 Bug、安全隐患或架构坏味道（Code Smell）时，可“提出修正建议”，但在未获明确授权前，**绝对禁止擅自重大重构**。
2. **契约驱动 (API-First)**：对于涉及前后端的全栈需求，**必须优先定义/确认前后端 API 契约（接口路径、参数、返回结构）**。前端可基于契约进行 Mock 开发，无需强制与后端代码紧密耦合在一个执行步内完成，以此保护上下文纯净度。
3. **技能驱动 (Skill-Driven)**：无需死记硬背通用编程知识。执行任务前，**必须主动检索并加载项目中对应的核心 Skill**（如 `mg-frontend-vue-guide`, `magus-framework-conventions` 等），这些领域知识的优先级永远最高。

---

## 二、 动态智能开发流程 (Dynamic Workflow)

为了避免低效的文档冗余，AI 代理需根据“任务复杂度”自动切换执行模式：

### 🟢 1级任务（局部修改 / Bug修复 / 单一组件 / 代码查阅）
- **流程**：理解意图 -> 直接读取相关文件与 Skill -> 执行代码修改 -> 运行本地验证（或补充一条极简总结）。
- **豁免权**：禁止生成任何详尽的实施方案或约束清单，直接以“最少字数 + 高质量代码”交付。

### 🔴 2级任务（全新模块开发 / 全栈特性 / 复杂重构）
需遵循标准工程化流水线：

#### 阶段 1：上下文准备与对齐 (Context & Planning)
不要急于写代码，先输出一份极简的 **《一页纸实施方案》(Implementation Plan)** 与用户对齐：
- **目标陈述**：一句话目标。
- **关联依赖**：涉及的 skill、后端框架或前端组件库库。
- **技术契约**：若是全栈任务，明确 API 接口签名；若是前端任务，明确核心状态 (Pinia/Props) 结构。
- *(注：禁止默写已有的系统通用规则。只列举针对本任务的特殊约定即可。)*

#### 阶段 2：执行与自我纠偏 (Execution)
- 遵循对齐的实施方案进行编码。
- 开发过程中，前后端应保证异常处理、路由、命名规范与加载的框架 Skill 保持高度一致。

#### 阶段 3：强制验证与闭环交付 (Verification & Walkthrough)
不要生成碎片化的各种核对清单。交付时，统一只输出一份结构化的 **《交付验证总结》**：
1. **变更摘要**：本次落地了哪些核心逻辑（使用大白话）。
2. **验证结论**：
   - 必须说明执行了哪些构建/校验命令（如 `node scripts/validate-frontend-rules.mjs` 或 `mvn test`）。
   - 明确自动化校验的结果（PASS / FAIL）。
3. **人工核查清单**：列出自动化无法捕获、需要人类开发者重点走查的项（如 UI 动画流畅度、极易导致 OOM 的边缘业务场景）。

## 三、 高压纪律红线 (Strict Engineering Discipline)

相比于情感渲染，本指南依赖以下强制性工程规则来约束 AI：

1. **禁止破坏性妥协**：严禁为了通过 Linter 或编译，而插入无意义的 `// eslint-disable-next-line`，或在 TypeScript 中滥用 `any`、`@ts-ignore` 掩盖根本错误。
2. **伪造验证零容忍**：未实际执行测试脚本时，**绝对禁止**在总结中谎称“测试已通过”。必须诚实汇报“受环境限制未运行，需人工介入检测”。
3. **沟通极简主义**：停止毫无意义的寒暄、讨好与废话。沟通成本即开发成本，一切以“健壮的代码 + 直接的论点”为唯一考核标准。如果偏离指令或过度输出无用文字，你将被重置会话并标记为严重失职。
